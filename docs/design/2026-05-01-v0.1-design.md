# moodle-student-mcp v0.1 design

Date: 2026-05-01

## 1. Overview

A Model Context Protocol server that lets a student talk to an LLM about their Moodle coursework. Read-only. The server exposes nine tools that call the Moodle Web Service REST API on behalf of the authenticated student, using a personal Web Service token.

This is the student-side counterpart to teacher-focused Moodle MCP servers. The reference repo `peancor/moodle-mcp-server` exposes teacher actions like listing students in a course or grading submissions. This server exposes student views: my courses, my upcoming deadlines, my grades, my quiz attempts.

### Goals
- Cover the most common student questions with one tool call each.
- Be a clean first OSS repo: tests, CI, lint, contribution docs, npm publishable.
- Keep the surface small. Nine read-only tools.

### Non-goals for v0.1
- No write actions. No assignment submission, no forum posting.
- No retries, caching, or rate limiting.
- No live Moodle integration tests in CI.
- No OAuth. Web Service token only.

## 2. Tech stack

- TypeScript, strict mode, ES2022 target, NodeNext module resolution.
- ESM only. Engines `node >= 18`.
- `@modelcontextprotocol/sdk` for the MCP server and stdio transport.
- Native `fetch` for HTTP. No axios.
- `zod` for tool input validation.
- `vitest` for tests, `msw` v2 for HTTP mocking.
- `eslint` with `@typescript-eslint`, `prettier`.
- Build: `tsc` direct to `dist/`. No bundler.
- Package manager: `npm`.

## 4. Components

Four layers, each with one responsibility. Outside in:

### 4.1 Entry point: `src/index.ts`

Loads config, builds `MoodleClient`, calls `getSiteInfo()` once to validate the token and resolve `userId`, builds the `McpServer`, registers every tool from the registry, attaches the stdio transport, starts the server. Exits non-zero with a friendly stderr message if config is missing or the token call fails.

### 4.2 Tools: `src/tools/<tool>.ts`

One file per tool. Each file exports a single object:

```ts
export const listCoursesTool = {
  name: 'list_courses',
  description: 'List all courses the authenticated student is enrolled in.',
  inputSchema: z.object({}),
  handler: async (input, ctx) => { ... },
};
```

`ctx` carries `{ client: MoodleClient, userId: number }`. Handlers validate input via the Zod schema, call one or two domain functions, shape the result, and return MCP text content (`{ content: [{ type: 'text', text: JSON.stringify(...) }] }`). `tools/index.ts` exports the array of all nine tool objects.

### 4.3 Domain wrappers: `src/moodle/<domain>.ts`

One file per Moodle subsystem: `courses`, `assignments`, `grades`, `calendar`, `quizzes`. Each exposes named functions like `getMyCourses(client, userId)` or `getAssignmentSubmission(client, assignId, userId)` that return typed responses. These wrappers hide raw `wsfunction` names and Moodle's parameter quirks (snake_case keys, array index suffixes like `courseids[0]`) behind clean TypeScript signatures.

### 4.4 HTTP wrapper: `src/moodle/client.ts`

The core `MoodleClient`. Single public method:

```ts
async call<T>(wsfunction: string, params: Record<string, unknown>): Promise<T>
```

Builds the URL `${baseUrl}/webservice/rest/server.php`, POSTs `application/x-www-form-urlencoded` with `wstoken`, `wsfunction`, `moodlewsrestformat=json`, and the flattened params. Parses JSON. Detects Moodle's "200 OK with error body" pattern (response has an `exception` field) and throws `MoodleApiError`. Throws `MoodleApiError` for non-2xx HTTP too.

### 4.5 Config: `src/config.ts`

Reads `MOODLE_URL` and `MOODLE_TOKEN` from `process.env`. Trims trailing slash on URL. Optional `MOODLE_DEBUG=1` flag for logging. Throws `ConfigError` with a clear message if anything required is missing.

### 4.6 Errors: `src/errors.ts`

Two error classes:
- `ConfigError(message)`: thrown by `config.ts` when env vars are missing or malformed.
- `MoodleApiError(code, message, context?)`: where `code` is either the synthetic `'NETWORK'` / `'HTTP_ERROR'`, or a Moodle errorcode string (e.g. `'invalidtoken'`). `context` is an optional object with `{ wsfunction?: string, httpStatus?: number, originalMessage?: string }` used for debug logging.

Tool handlers catch these and return MCP error responses (`isError: true`) instead of crashing the server.

## 5. Tools

All inputs validated with Zod. All outputs returned as a single text content with pretty-printed JSON. Tool input keys use `snake_case` to match Moodle conventions.

### 5.1 `list_courses`
- Description: "List all courses the authenticated student is currently enrolled in."
- Input: none.
- Calls: `core_enrol_get_users_courses(userid)`. The userid is cached from startup.
- Output: array of `{ id, fullname, shortname, summary, startdate, enddate, progress, lastaccess }`.

### 5.2 `get_course_contents`
- Description: "Get the structure of a course (sections, modules, files, links, descriptions). Use this to see what content and resources are inside a course."
- Input: `{ course_id: number }`.
- Calls: `core_course_get_contents(courseid)`.
- Output: sections array, each with `{ id, name, summary, modules: [{ id, name, modname, url, description, contents: [...] }] }`.

### 5.3 `list_assignments`
- Description: "List assignments across the student's courses with due dates and basic info."
- Input: `{ course_ids?: number[], include_submission_status?: boolean }` (defaults: all enrolled, false).
- Calls: `mod_assign_get_assignments(courseids)`. If `include_submission_status` is true, also calls `mod_assign_get_submission_status(assignid, userid)` per assignment in parallel.
- Output: flat array of `{ course_id, course_name, assignment_id, name, intro, duedate, allowsubmissionsfromdate, gradingduedate, submission_status?, has_submitted? }`.

### 5.4 `get_assignment`
- Description: "Get full details for one assignment plus the student's own submission status, files, grade, and feedback."
- Input: `{ assignment_id: number, course_id: number }`. The course id keeps the call cheap. The LLM gets it from `list_courses` or `list_assignments`.
- Calls: `mod_assign_get_assignments(courseids: [course_id])` filtered to the assignment, plus `mod_assign_get_submission_status(assignid, userid)`.
- Output: `{ assignment: { id, name, intro, duedate, ... }, submission: { status, files, grade, feedback } }`.

### 5.5 `get_grades`
- Description: "Get the student's grade report for one course or all enrolled courses."
- Input: `{ course_id?: number }`.
- Calls: `gradereport_user_get_grade_items(courseid, userid)`. If `course_id` is omitted, calls `core_enrol_get_users_courses` first, then fans out one parallel call per course.
- Output: per-course array of `{ course_id, course_name, items: [{ itemname, gradeformatted, percentageformatted, feedback, weight }] }`.

### 5.6 `list_upcoming_events`
- Description: "List upcoming calendar events for the student over the next N days. Includes both action items and general events (class meetings, school events)."
- Input: `{ days_ahead?: number }` (default 14, range 1..90).
- Calls: `core_calendar_get_calendar_upcoming_view()`. Filtered client-side by `timestart <= now + days_ahead * 86400`.
- Output: array of `{ id, name, type, course_id?, course_name?, timestart, timestart_iso, description }`.

### 5.7 `list_upcoming_deadlines`
- Description: "List the student's upcoming action items (assignment submissions, quiz attempts, lesson deadlines) sorted by due date. Matches the Moodle dashboard Timeline."
- Input: `{ days_ahead?: number, limit?: number }` (defaults: 14 days, 25 events).
- Calls: `core_calendar_get_action_events_by_timesort` with `timesortfrom = now`, `timesortto = now + days_ahead * 86400`, `limitnum = limit`.
- Output: array of `{ id, name, course_id, course_name, modulename, instance, timestart, timestart_iso, action: { name, url, itemcount }, description }`. The `action` object captures the to-do nature of the event (e.g., `name: "Submit assignment"`, `itemcount: 1`).

### 5.8 `list_quizzes`
- Description: "List quizzes across the student's courses, with open/close times and attempt limits."
- Input: `{ course_ids?: number[] }` (defaults to all enrolled).
- Calls: `mod_quiz_get_quizzes_by_courses(courseids)`. If no course_ids, calls `core_enrol_get_users_courses` first.
- Output: array of `{ course_id, course_name, quiz_id, name, intro, timeopen, timeclose, attempts_allowed, time_limit, grade }`.

### 5.9 `get_quiz_attempts`
- Description: "Get the student's attempts for a specific quiz, with scores and timestamps."
- Input: `{ quiz_id: number }`.
- Calls: `mod_quiz_get_user_attempts(quizid, userid)`.
- Output: array of `{ id, attempt, state, timestart, timefinish, sumgrades, grade }`.

### 5.10 Cross-cutting decisions
- Tool input keys use `snake_case` (matches Moodle conventions and reads naturally to the LLM).
- Output keys preserve Moodle field names where they are clear, with light enrichment (joining `course_name`, formatting Unix timestamps as ISO alongside the raw value).
- All nine tools are read-only. No write actions in v0.1, which avoids any risk of the LLM accidentally submitting an assignment.

## 6. Configuration and auth

### 6.1 Environment variables
- `MOODLE_URL` (required): base URL of the Moodle instance, e.g. `https://moodle.myschool.edu`.
- `MOODLE_TOKEN` (required): personal Moodle Web Service token.
- `MOODLE_DEBUG` (optional): set to `1` to log WS calls (URL, function, params, response time) to stderr.

### 6.2 Startup auth check
On server start, before registering tools, the server calls `core_webservice_get_site_info` once. This validates that the token works and caches `userId` and `sitename` for later calls. Failure here is fatal: the server prints a friendly error to stderr and exits non-zero. The cached `userId` is passed into every tool handler context.

## 7. Error handling

Errors flow through four layers:

1. Network errors (DNS, timeout, refused) throw `new MoodleApiError('NETWORK', friendlyMessage, { wsfunction })`.
2. HTTP non-2xx (rare, since Moodle returns 200 even for app errors) throw `new MoodleApiError('HTTP_ERROR', message, { wsfunction, httpStatus })`.
3. Moodle returns 200 OK with `{ exception, errorcode, message, debuginfo? }` for application errors. The client detects the `exception` field and throws `new MoodleApiError(errorcode, friendlyMessage, { wsfunction, originalMessage })`.
4. Friendly mappings in `errors.ts` for the most common error codes:
   - `invalidtoken`, `accessexception`: "Your Moodle token is invalid or expired. Generate a new token in your Moodle profile and update MOODLE_TOKEN."
   - `accessdenied`, `nopermissions`: "Your token does not have permission for this action. Make sure the token's external service includes the required Moodle Web Service functions."
   - `dml_missing_record_exception`: "The requested item was not found. Check the ID."
   - Anything else: pass through the Moodle message.

Tool handlers wrap their body in try/catch. On `MoodleApiError` or `ZodError`, they return an MCP error response (`isError: true`) with the friendly message. The server stays alive; only the failing tool call surfaces an error to the LLM. Anything unexpected is caught at the server boundary, logged to stderr with stack, and returned as a generic "Internal error in moodle-student-mcp" so the LLM can react sensibly.

### 7.1 Out of scope for v0.1
- No retry/backoff. Moodle does not publish rate limits. If we see flakes in practice, add it later.
- No caching. Tools are read-only and Moodle is fast enough; caching adds correctness questions (stale grades) we should not solve preemptively.
- No token rotation or refresh. Moodle WS tokens are long-lived; users rotate manually.

### 8.3 What each layer tests
- `tests/moodle/client.test.ts`: form-encoding format; array flattening (`courseids[0]=1&courseids[1]=2`); error detection and mapping; network failure handling; debug logging.
- `tests/moodle/<domain>.test.ts`: each wrapper sends the right `wsfunction` with the right params, returns a typed response shape matching the fixture.
- `tests/tools/<tool>.test.ts`: input validation rejects bad inputs (e.g., `get_course_contents` without `course_id`); handler calls the right WS endpoints; the JSON content the LLM sees has the expected shape; error path returns an MCP `isError: true` response.
- `tests/config.test.ts`: missing env vars produce useful errors; URL trailing slash gets trimmed.

### 8.4 Coverage
Goal 85% on `src/moodle/` and `src/tools/`. `src/index.ts` (wiring) is excluded; it is integration-tested indirectly when the package is run by an MCP host.

### 8.5 Out of scope for v0.1
No live Moodle integration tests in CI. `CONTRIBUTING.md` documents a manual smoke-test procedure for contributors who want to verify against a real Moodle instance.

GitHub Actions workflow at `.github/workflows/ci.yml`:
- Triggers: push to `main`, PRs against `main`.
- Matrix: Node 18, 20, 22 on `ubuntu-latest`.

After the first manual `0.1.0` publish, add `.github/workflows/release.yml` triggered on tag push `v*` that runs the same checks then `npm publish --access public`. Requires an `NPM_TOKEN` repo secret. Holding off in v0.1 keeps publishing simple while the package is finding its footing.
