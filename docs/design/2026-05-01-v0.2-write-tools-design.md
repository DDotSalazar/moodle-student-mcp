# moodle-student-mcp v0.2 design: write tools

Date: 2026-05-01

## 1. Overview

v0.2 adds eleven write-side tools so the student can do things in Moodle through an LLM, not just read. Submitting an assignment, taking a quiz, posting in a forum, sending a message, voting in a course poll, filling in a feedback form, and creating a personal calendar event are all covered.

The architecture from v0.1 stays the same. New tools live alongside the existing nine in `src/tools/`. New domain wrappers go under `src/moodle/` (e.g. `forum.ts`, `messages.ts`, `feedback.ts`). The `MoodleClient` is unchanged, but a small additional surface is needed for the file-upload endpoint (`/webservice/upload.php`), which uses `multipart/form-data` instead of the form-encoded WS endpoint.

### Goals
- Mirror Moodle's natural student-side write surface with one tool per intent.
- Match v0.1's testing rigor: every new tool gets unit tests, every new domain wrapper gets unit tests, fixtures captured from real responses.
- Keep ergonomics: tool inputs are flat and small, outputs are pretty-printed JSON, no hidden state.
- Make the unsafe parts loud: writing tools name the action they take and never silently escalate (e.g. saving a draft never auto-finalises).

### Non-goals for v0.2
- No `mod_workshop`, `mod_data`, `mod_glossary`, `mod_scorm`, `mod_h5p`, `mod_lesson`. Niche modules deferred.
- No `enrol_self_enrol_user`. Out of scope for a study helper.
- No automatic OCR or fallback for image-only quiz questions. Tools return image URLs (auth-tokenised) and let the LLM client render them.
- No retries, no caching, no rate limiting (same as v0.1).
- No live integration tests in CI. Manual smoke procedure documented in `CONTRIBUTING.md` will be extended for write tools.
- No transactional rollback if a multi-step tool fails mid-way (e.g. file uploaded but submission rejected). The tool surfaces the partial state and the LLM decides what to do.

## 2. Tools

Eleven new tools. Names use `snake_case` to match v0.1.

| Tool | Purpose |
|---|---|
| `submit_assignment` | Save draft and optionally finalise an assignment submission (text and/or files) |
| `start_quiz_attempt` | Start or resume an attempt; returns the attempt id and normalised questions |
| `save_quiz_answers` | Save answers to one or more questions in an in-progress attempt |
| `submit_quiz_attempt` | Finalise an attempt for grading |
| `start_forum_discussion` | Start a new discussion thread in a forum |
| `post_forum_reply` | Reply to an existing forum post |
| `send_message` | Send a direct message to another Moodle user |
| `mark_messages_read` | Mark one or more messages as read |
| `submit_choice_response` | Submit a response to a `mod_choice` poll |
| `submit_course_feedback` | Submit a response to a `mod_feedback` form |
| `create_calendar_event` | Create a personal calendar event |

### 2.1 `submit_assignment`

- Description: "Save a draft submission for an assignment, with optional online text and file attachments. Pass `submit_for_grading: true` to finalise after saving."
- Input:
  ```ts
  {
    assignment_id: number,
    text?: string,             // online text submission body
    text_format?: 'html' | 'plain', // default 'html'
    files?: string[],          // local absolute paths
    submit_for_grading?: boolean // default false; saves draft only
  }
  ```
- Calls:
  1. For each file in `files`, POST `multipart/form-data` to `/webservice/upload.php?token=...` and capture the returned `itemid`. All files share one `itemid` (Moodle's draft area model).
  2. `mod_assign_save_submission` with `plugindata`:
     - `onlinetext_editor: { text, format: 1|0, itemid: 0 }` if `text` provided.
     - `files_filemanager: <itemid>` if any files uploaded.
  3. If `submit_for_grading`, `mod_assign_submit_for_grading(assignmentid, acceptsubmissionstatement: 1)`.
- Output: `{ saved: true, finalised: boolean, files_uploaded: number, warnings: [] }`. Warnings from each WS call are concatenated.
- Errors: file path missing, file too large for the assignment, assignment closed, no online-text plugin enabled, missing required text or files. Surfaced as `MoodleApiError` or local `ConfigError` for missing files.

### 2.2 `start_quiz_attempt`

- Description: "Start a new attempt at a quiz, or resume the in-progress attempt if one exists. Returns the attempt id and a normalised list of questions across all pages."
- Input: `{ quiz_id: number }`.
- Calls:
  1. `mod_quiz_get_user_attempts(quizid, userid, status: 'unfinished')`. If a non-empty attempt exists, use it; otherwise call `mod_quiz_start_attempt(quizid)`.
  2. For each page (start with page 0, then walk pages until `nextpage = -1`), call `mod_quiz_get_attempt_data(attemptid, page)`.
  3. Parse each question's `html` to extract the structured form (see Section 3 for per-type encoding).
- Limitation: password-protected quizzes (`mod_quiz_start_attempt` rejects without `preflightdata`) are not supported in v0.2. The tool surfaces Moodle's `passwordrequired` errorcode unchanged so the LLM can tell the user to take it manually. Preflight support is in Section 13.
- Output:
  ```ts
  {
    attempt_id: number,
    quiz_id: number,
    state: 'inprogress' | 'overdue',
    timestart: number,
    time_limit_seconds?: number,
    questions: Array<{
      slot: number,
      type: 'multichoice' | 'truefalse' | 'shortanswer' | 'essay'
          | 'numerical' | 'match' | 'ddwtos' | 'unsupported',
      text: string,           // plain text
      html: string,           // original Moodle HTML
      images: Array<{ src: string, alt?: string }>, // auth-tokenised URLs
      options?: Array<{ index: number, text: string, html: string }>, // multichoice/match
      multiselect?: boolean,  // multichoice: true if checkboxes (multi), false if radios (single)
      sub_slots?: Array<{ index: number, prompt: string }>, // match/ddwtos
      number_unit_required?: boolean, // numerical
      max_length?: number,    // essay
      raw_form_fields: Record<string, string> // captured for save_quiz_answers
    }>
  }
  ```
- The `raw_form_fields` field is the trick: the LLM never has to look at it, but `save_quiz_answers` needs the original Moodle field names (`q$slot:$qid_answer`, etc.) to reconstruct the data array Moodle expects. The tool stores them in the response so the LLM can echo them back; alternative would be server-side state, which we explicitly avoid.

### 2.3 `save_quiz_answers`

- Description: "Save answers to one or more questions in an in-progress quiz attempt. Does not finalise the attempt; can be called repeatedly to update or add answers."
- Input:
  ```ts
  {
    attempt_id: number,
    answers: Array<{
      slot: number,
      raw_form_fields: Record<string, string>, // from start_quiz_attempt
      value: AnswerValue // see Section 3 per-type
    }>
  }
  ```
- Calls: `mod_quiz_save_attempt(attemptid, data)`. The handler converts each `(raw_form_fields, value)` pair into the flat `data: [{name, value}]` array Moodle requires (Section 3).
- Output: `{ saved: true, slots: number[], warnings: [] }`.

### 2.4 `submit_quiz_attempt`

- Description: "Finalise an in-progress quiz attempt for grading. After this, no further answers can be saved."
- Input: `{ attempt_id: number }`.
- Calls: `mod_quiz_process_attempt(attemptid, data: [], finishattempt: 1, timeup: 0)`.
- Output: `{ finalised: true, state: string, sumgrades?: number, grade?: number }`. Grade is included if Moodle returns it immediately (some quizzes auto-grade, others defer until manually marked).

### 2.5 `start_forum_discussion`

- Description: "Start a new discussion thread in a forum."
- Input:
  ```ts
  {
    forum_id: number,
    subject: string,
    message: string,
    message_format?: 'html' | 'plain', // default 'html'
    pin?: boolean,                     // teacher only; surfaced for completeness
    discussion_type?: 'general' | 'qanda', // forum-dependent
  }
  ```
- Calls: `mod_forum_add_discussion(forumid, subject, message, options?)`. Options translate to Moodle's `[{name, value}]` array.
- Output: `{ discussion_id: number, warnings: [] }`.

### 2.6 `post_forum_reply`

- Description: "Reply to an existing forum post."
- Input:
  ```ts
  {
    post_id: number,           // the post being replied to
    subject: string,
    message: string,
    message_format?: 'html' | 'plain'
  }
  ```
- Calls: `mod_forum_add_discussion_post(postid, subject, message, options?)`.
- Output: `{ post_id: number, warnings: [] }`.

### 2.7 `send_message`

- Description: "Send a direct message to another Moodle user."
- Input: `{ to_user_id: number, text: string }`.
- Calls: `core_message_send_instant_messages([{ touserid, text, textformat: 1 }])`.
- Output: `{ message_id: number, sent_at_iso: string }`.

### 2.8 `mark_messages_read`

- Description: "Mark one or more messages as read."
- Input: `{ message_ids: number[] }`.
- Calls: `core_message_mark_message_read(messageid, timeread?)` per id, in parallel.
- Output: `{ marked: number, failed: number[] }`.

### 2.9 `submit_choice_response`

- Description: "Submit a response to an in-course poll (`mod_choice`). Replaces any prior response."
- Input: `{ choice_id: number, response_ids: number[] }` (multiple ids only if the choice allows multi-select).
- Calls: `mod_choice_submit_choice_response(choiceid, responses)`.
- Output: `{ submitted: true, answers: number[], warnings: [] }`.

### 2.10 `submit_course_feedback`

- Description: "Submit responses to a feedback form. Feedback can span multiple pages, so this tool walks pages internally and submits all responses in one call."
- Input:
  ```ts
  {
    feedback_id: number,
    responses: Array<{ item_id: number, value: string | number | string[] }>
  }
  ```
- Calls:
  1. `mod_feedback_get_items(feedbackid)` to learn the item layout.
  2. `mod_feedback_process_page(feedbackid, page, responses)` for each page, advancing until `completed: true`.
- Output: `{ submitted: true, completion_page_contents?: string }`.

### 2.11 `create_calendar_event`

- Description: "Create a personal calendar event for the authenticated student."
- Input:
  ```ts
  {
    name: string,
    description?: string,
    timestart: string,        // ISO datetime, converted to epoch internally
    duration_seconds?: number, // default 0 (point-in-time event)
    course_id?: number        // if attached to a course (rare for personal events)
  }
  ```
- Calls: `core_calendar_create_calendar_events([{ name, description, eventtype: 'user', timestart, timeduration, ... }])`.
- Output: `{ event_id: number }`.

## 3. Quiz answer encoding

Moodle's quiz API takes answers as a flat `data: [{name, value}]` array. The names embed the question slot and id (`q1:42_answer`) and the format depends on the question type. v0.2 supports the eight types below; anything else is returned with `type: 'unsupported'` and `save_quiz_answers` rejects it with a clear error.

For each row, `slot` and `qid` come from the form HTML returned by `mod_quiz_get_attempt_data` and are captured into `raw_form_fields` by `start_quiz_attempt`.

The encoder dispatches on the parsed question's `type`, plus `multiselect` for multichoice.

| Question type | Input value | Encoded data fields |
|---|---|---|
| `multichoice`, `multiselect: false` | `option_index: number` (or `-1` to clear) | `q{slot}:{qid}_answer = option_index` |
| `multichoice`, `multiselect: true` | `option_indices: number[]` | `q{slot}:{qid}_choice{i} = 0\|1` for each option |
| `truefalse` | `value: boolean` | `q{slot}:{qid}_answer = "1"\|"0"` |
| `shortanswer` | `text: string` | `q{slot}:{qid}_answer = text` |
| `essay` | `{ text: string, format?: 'html'\|'plain' }` | `q{slot}:{qid}_answer = text`, `q{slot}:{qid}_answerformat = 1\|0` |
| `numerical` | `{ value: string, unit?: string }` | `q{slot}:{qid}_answer = "{value}{unit?}"` |
| `match` | `pairs: Array<{sub_slot: number, stem_index: number}>` | `q{slot}:{qid}_sub{sub_slot} = stem_index` |
| `ddwtos` | `placements: Array<{drop_slot: number, draggable_index: number}>` | `q{slot}:{qid}_p{drop_slot} = draggable_index` |

Encoding lives in `src/moodle/quiz_answers.ts` as a pure function `encode(question, value): Array<{name, value}>`. It is unit-tested per type with handcrafted question fixtures so the encoder is locked down independently from the quiz tools.

### Detection

Question type is detected from the parsed `html` in `mod_quiz_get_attempt_data`. Moodle uses a `qtype_<type>` class on the outer wrapper (e.g. `<div class="que multichoice">`). The parser looks at the first matching `qtype_*` class and falls back to `unsupported`.

### Image handling

Question HTML can contain `<img src="...pluginfile.php/...">` references. These require `?token=<MOODLE_TOKEN>` appended to load. The parser rewrites image src URLs to include the token, then exposes them in `images: [{ src, alt? }]`. The LLM client (Claude Desktop, etc.) is responsible for fetching them and rendering them inline as MCP image content blocks. If the client cannot, the image stays a URL the user can open manually. v0.2 does not OCR.

## 5. File upload

`mod_assign_save_submission` references attached files indirectly: the file goes up via `/webservice/upload.php` first, returning a draft `itemid` that the WS call references. v0.2 handles this in two places:

- `MoodleClient.uploadFiles(files: string[]): Promise<{ itemid: number; uploaded: Array<{filename, filesize}> }>`. Builds a multipart request, posts to `/webservice/upload.php?token=...&filearea=draft&itemid=0`. All files in one call share an `itemid` (Moodle assigns one if `itemid=0` is sent).
- `submit_assignment` calls this once, then plugs the resulting `itemid` into `plugindata.files_filemanager`.

This is the only place v0.2 touches a Moodle endpoint outside the WS REST path. It lives on `MoodleClient` so unit tests can mock it the same way (msw v2 handles multipart fine).

## 6. Errors

Same model as v0.1. New error mappings to add to `errors.ts`:

| Moodle errorcode | Friendly message |
|---|---|
| `notopenforsubmissions` | "This assignment is not open for new submissions." |
| `submissionsclosed` | "Submissions are closed for this assignment." |
| `nopermissiontoaddpost` | "You do not have permission to post in this forum." |
| `cannotsendmessage` | "Could not send the message. The recipient may have messaging disabled or have blocked you." |
| `attemptalreadyclosed` | "This quiz attempt has already been finalised." |
| `quizalreadystarted` | "There is already an in-progress attempt at this quiz; resume it instead." |
| `feedback_completed` | "You have already completed this feedback." |
| `cannotsavetempfile` | "File upload failed on Moodle's side. Try again." |

Tool handlers wrap calls in try/catch and return MCP `isError: true` responses, same as v0.1. Multi-step tools (`submit_assignment`, `submit_course_feedback`, `start_quiz_attempt`) report partial success: e.g. "files uploaded, save_submission failed" rather than rolling back. Rollback is not feasible (Moodle has no transactional API) and silent retry is dangerous.

## 7. Safety patterns

Three rules apply to every write tool.

1. **Default to non-destructive.** `submit_assignment` saves a draft unless the caller passes `submit_for_grading: true`. `submit_quiz_attempt` is a separate tool from `save_quiz_answers`. `submit_choice_response` overwrites prior responses but always reports the previous response in the output so the LLM can show it in the next message.
2. **Surface what was actually done.** Output keys use past-tense verbs (`saved`, `finalised`, `submitted`, `marked`) to make it impossible to confuse "ran the tool" with "took the action". The `submit_for_grading` and `finalised` fields are explicit booleans.
3. **No auto-retry.** A failed write returns the error and stops. Retrying writes is the LLM's call after seeing the error, not the tool's.

The README disclaimer (Section 9) restates these from the user's side.

## 8. Configuration changes

No new env vars. The same `MOODLE_TOKEN` is used for write operations; the token's underlying Web Service must include the new functions listed in Section 11. `docs/moodle-setup.md` is updated to list every WS function v0.2 uses, so admins can grant a minimal custom service if `moodle_mobile_app` is locked down.

### 10.3 What gets tested

- **Encoders** (`quiz_answers.test.ts`): every supported question type round-trips through `encode(question, value)` and produces the exact `{name, value}` array Moodle expects. Hand-built question fixtures cover edge cases: empty multichoice (clear), single-vs-multi multichoice, numerical with and without unit, match with missing pairs, ddwtos with the same draggable used twice.
- **Question parser** (`quiz_questions.test.ts`): given known `mod_quiz_get_attempt_data` HTML payloads, the parser extracts the right type, options, sub-slots, and `raw_form_fields`. Image rewriting adds `?token=` to all `pluginfile.php` URLs.
- **Multi-step tools** (`submit_assignment.test.ts`, `start_quiz_attempt.test.ts`): partial-failure paths. If files upload but `save_submission` errors, the tool reports `files_uploaded > 0, saved: false` rather than crashing. If page 2 of a quiz fetch fails, the tool reports the questions it already gathered.
- **Safety paths**: `submit_assignment` with `submit_for_grading: false` never calls `mod_assign_submit_for_grading`. `save_quiz_answers` rejects an `unsupported` question type with a clear error.
- **Standard layer tests**: input validation, snake_case keys, output shape, Moodle errorcode mapping.

### 10.4 Coverage

Goal stays 85% on `src/moodle/` and `src/tools/`. v0.2 raises the line count enough that the threshold should be checked separately for the new code; CI's `vitest --coverage` numbers are reported per-file already, so a regression is visible without changing the global target.

### 10.5 Manual smoke procedure

`CONTRIBUTING.md` gets a new section walking through the write tools against `sandbox.moodledemo.net` (student credentials in the README of moodledemo). Each tool has a one-line command using the package's `dev` mode and a hand-rolled stdin JSON-RPC frame. The procedure is opt-in for contributors; CI does not run it.
